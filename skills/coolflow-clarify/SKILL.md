# 需求澄清（clarify）

把模糊需求**澄清**成清晰、无歧义、可直接作为 design 阶段输入的需求。
本阶段采用**「先一次性收集问题，再多轮对话问答」**：先把所有疑问点收集齐，
集中抛给用户，通过对话让用户回答；用户回答引出的次生问题继续追问，直到全部清晰。
澄清完成后，把整个问答过程记录到产物文档的「问题澄清记录」部分。

## 阶段产物落点（遵守元技能「阶段产物落点规则」）
- 产物文档：`coolflow/<需求名称>/clarify-<YYYYMMDD>.md`
- `coolflow/`、`<需求名称>/` 不存在则创建；`<需求名称>` 从用户输入获取，**取不到就直接问用户**。

## 产物文档结构（固定两大部分）
```markdown
# 需求澄清：<需求名称>
- 日期 / 原始输入摘要

## 问题澄清记录
（按轮次记录对话澄清过程：每个问题 + 用户的回答；次生问题归入其触发的轮次）
### 第 1 轮
- Q1：<疑问点> → A：<用户回答>
- Q2：<疑问点> → A：<用户回答>
### 第 2 轮（由上轮回答引出的次生问题）
- Q3：<疑问点> → A：<用户回答>
...

## 需求澄清结果
（所有问题澄清后，在此写入最终的、可作为 design 输入的清晰需求）
```

## 流程清单（请为每步建 TodoWrite）

### 步骤 1：分析输入（文字 / 链接 / 文字+链接）
- 解析用户输入。**若含链接**：用 `web-access` 技能抓取链接内容后再分析（需求文档、原型、设计稿等）。
- **检索项目知识库辅助理解（渐进式披露：先索引后细读，绝不预加载整库）**：尝试从知识库找能帮助
  精确理解本需求的既有知识（业务概念/术语、领域规则、状态语义、API 契约等），用来校正对需求的理解、
  梳理实体与状态的逻辑关系、发现隐含约束与潜在冲突。严格按下面顺序，避免把不相干内容读进上下文：
  1. **定位（不写死目录，先问项目自己）**：不同项目知识库的组织方式不同，**不要假设固定的子目录名**。
     先尝试 Read 项目的 `PROJECT_RULES/CLAUDE.md`（即 `<项目>/.claude/CLAUDE.md`，大小写依项目而定，
     也可能是 `claude.md`），从中确认知识库的**具体位置与组织约定**；据此定位知识库根。
     该文件不存在或未说明，再到 `PROJECT_RULES/` 下就近探查；仍无则用 `DEFAULTS/knowledge-base/` 兜底。
  2. **只读索引/标题，先建清单**：在定位到的知识库目录下，用 Glob 探查**实际**的目录结构与文件名
     （以项目真实结构为准，不臆造目录）；若有 `_index.md` 之类索引文件先读它，必要时只读各文件标题/小节，
     建一张「主题 → 文件」清单，**此阶段不加载任何正文**。
  3. **按相关性细读**：仅当某文件主题与本需求**明确相关**时，才 Read 它的正文；不相关的一律不读。
  4. 知识库不存在或无相关条目 → 跳过，照常分析（本步是尽力而为，不阻塞澄清）。
- 提炼背景、目标、用户场景，标出所有不明确处与隐含假设（结合上一步检索到的知识库知识一起判断）。

### 步骤 2：确定需求名称
- 与用户确认 `<需求名称>`（取不到就问）。它用于产物目录。

### 步骤 3：一次性收集本轮所有疑问点
- 把当前能想到的**所有**需澄清的问题收集齐（不要挤牙膏式一个个问），编号成一组。

### 步骤 4：通过对话集中提问（多轮）
- 在**对话中**一次性把这组问题抛给用户（编号列出），请用户逐条回答。
- 这是普通多轮对话，**不需要用户去编辑文档**。

### 步骤 5：处理回答与次生问题（循环到清晰）
- 用户回答后，判断是否仍有不清晰处、或回答**引出了新的次生问题**。
- 若有 → 把次生/遗留问题再次集中抛给用户对话追问。
- **循环直到需求清晰无误、足以作为 design 阶段输入。**

### 步骤 6：记录问答过程 + 写澄清结果
- 把整个多轮问答过程按轮次写入产物文档「问题澄清记录」（每条含 问题 → 用户回答）。
- 在「需求澄清结果」写入最终的、结构化的清晰需求，至少含：
  需求描述、用户故事、边界情况、影响范围、验收标准。这是 design 阶段的唯一需求输入。

## 收尾
按 `coolflow-using-flow` 纪律调用 `coolflow-precipitate` 反思沉淀
（常见可沉淀项：澄清出来的业务约束 → knowledge-base 业务域）。

## 红旗表
| 念头 | 真相 |
|---|---|
| "想到一个问题就问一个" | 要先一次性收集齐所有问题再集中对话提问，减少来回轮次。 |
| "用户回答有歧义/引出新问题，先放着" | 次生问题必须继续追问，循环到全部清晰才能产出澄清结果。 |
| "对话问完就够了，不用写文档" | 必须把问答过程记入「问题澄清记录」，并写「需求澄清结果」。 |
| "链接我猜个大概内容" | 含链接必须用 web-access 抓取真实内容，不要猜。 |
| "知识库整个读一遍最稳妥" | 违反渐进式披露。先读 `_index` 与标题建清单，只 Read 与本需求明确相关的文件，别把无关内容塞进上下文。 |
| "还有点模糊，先交给 design 吧" | 必须循环到清晰无误才能产出「需求澄清结果」。 |
| "需求名称我随便起一个" | 取不到就问用户，不要自己编目录名。 |
