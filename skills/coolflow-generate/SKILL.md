# 代码生成（generate）

根据 design 阶段产物（技术方案文档），生成**可应用于生产环境的真实代码**，
代码必须符合**目标项目的对应规范**。本阶段采用**任务清单分步执行**，全程留痕。

## 硬门禁（开工前必过）

<HARD-GATE id="code-needs-design">
🚫 **没有「通过评审的技术方案文档」，禁止生成任何业务代码。**
无论需求看起来多简单。开工前必须确认：
  ① 存在 `coolflow/<需求名称>/design-<日期>.md`；
  ② 该方案已被用户在 design 环节分段确认。
两者缺一，停下，先回到 `coolflow-design` 技能。
**可跳过性**：仅当用户下达**精确跳过指令**（明确跳过用词 + 点名本门禁/设计阶段，如「跳过设计直接写」）才可跳过；
模糊话不算，须先追问确认一次。详见元技能 `coolflow-using-flow` 的「硬门禁跳过协议」。
</HARD-GATE>

## 阶段产物落点（遵守元技能「阶段产物落点规则」）
- **实现代码**：写进对应项目目录（符合该项目规范），**不在 coolflow 下**。
- **执行记录文档**：`coolflow/<需求名称>/generate-<YYYYMMDD>.md`（任务清单 + 每步关键信息）。

## 流程清单（请为每步建 TodoWrite）

### 步骤 1：认真分析需求内容
- Read `coolflow/<需求名称>/design-<日期>.md`，吃透技术方案的每个改动点。
- **加载目标项目规范（不写死目录，先问项目自己 + 渐进式披露）**：各项目规范的组织方式不同，
  **不要假设固定的规则目录/文件名**。
  1. **取索引**：先尝试 Read 项目的 `PROJECT_RULES/CLAUDE.md`（即 `<项目>/.claude/CLAUDE.md`，
     大小写依项目而定，也可能是 `claude.md`），从中获取规范的位置与索引；该文件不存在或未说明，
     再到 `PROJECT_RULES/` 就近探查，仍无则用 `DEFAULTS/global-rules/` 兜底。
  2. **按需细读**：据索引用 Glob/标题建一张「主题 → 规则文件」清单，**只 Read 与本次改动相关的规则**，
     不预加载无关规范（渐进式披露）。
  3. **兼顾项目天然规范**：结合 `PROJECT_ROOT` 的 `package.json` / `.eslintrc*` / `tsconfig.json` / 现有代码风格。
  - 优先级遵循元技能「分层规范解析」：项目专属 > 项目级通用/技术栈 > 项目天然规范 > 全局兜底。
- 总结：本次要改哪些文件、遵循哪些项目约定、依赖关系与执行顺序。生成代码必须符合**当前项目**规范。

### 步骤 2：划分任务步骤，列出任务清单
- **判断单测设施，按需询问是否补单测**：先探查项目是否已有单测设施——如测试框架配置
  （jest / vitest / mocha 等）、`__tests__/` 或 `*.test.*` / `*.spec.*` 文件、`package.json` 的 `test` 脚本与测试依赖。
  - **存在单测设施** → 询问用户：本次需求是否需要为改动点补充单测？用户同意，则把"编写单测"作为
    任务清单中的**独立条目**纳入（明确目标文件与覆盖的关键场景）；用户不要则不加。
  - **没有单测设施** → 不强加单测，但可在变更摘要里提示"项目暂无单测设施"。
- 把实现拆成**有序、可独立验证**的任务，写入 `generate-<YYYYMMDD>.md` 的「任务清单」：
  ```markdown
  ## 任务清单
  - [ ] 1. <任务> — 目标文件 / 验收点
  - [ ] 2. ...
  ```
- 同时用 TodoWrite 建对应条目。

### 步骤 3：一步接一步执行（每步打标记 + 留痕）
- **严格按清单顺序逐个执行**，不要跳步、不要并发铺开。
- 每完成一步：
  1. 生成/修改真实代码（生产级，符合项目规范，不留 TODO 占位糊弄）。
  2. **打标记**：把该步在「任务清单」里勾选 `- [x]`，TodoWrite 同步置 completed。
  3. **记录关键信息**到 `generate-<YYYYMMDD>.md` 的「执行记录」：
     ```markdown
     ### 步骤 N：<任务>（✅）
     - 改动文件：path:line
     - 关键决策 / 实现要点：
     - 遇到的问题与处理：
     - 自检结果：lint/type/test
     ```
- 进入下一步前，确保当前步可用（必要时做局部自检）。

### 步骤 4：整体自检与收尾
全部任务完成后，**先列出自检清单，再逐项执行，最后写收尾文档**——不要笼统地说"自检通过"。

> **跑命令前先确认正确用法**：执行任何需要跑命令的自检项（构建 / 类型检查 / lint / 测试）前，
> 先 Read 项目根目录的 `README.md`（辅以 `package.json` 的 `scripts`），找出该项目**正确的命令与用法**
> 再执行，避免用错命令导致失败、提高成功率。README 缺失或没写，再退回用 `package.json` scripts / 常规命令。

1. **列出自检清单**（按本次改动定制，至少覆盖以下项；用 TodoWrite 或清单形式落到 `generate-<YYYYMMDD>.md`）：
   - [ ] **Lint**：按项目 lint 配置检查改动文件，修掉新增的告警/错误。
   - [ ] **类型检查**：若项目用 TS，跑类型检查（如 `tsc --noEmit` 或项目脚本），确保无新增类型错误。
   - [ ] **相关测试**：
       - **项目有单测设施**：运行受本次改动影响的单测（优先只跑相关范围而非全量），确认通过；
         若步骤 2 中应用户要求新增了单测，先确保这些新单测覆盖关键场景并全部通过。
       - **无单测设施 / 本次未写单测**：对关键逻辑做**人工校验**——逐条核对边界条件、状态流转、
         异常分支，并与 design 的改动点一一对应；必要时在 dev 环境**冒烟验证主流程**。
   - [ ] **需求对齐**：逐条核对 design 里的改动点是否都已实现，无遗漏、无跑偏。
2. **逐项执行清单**：按上面每一项实际执行，记录结果（通过 / 失败 / 不适用 + 关键输出或命令）。
   - 出现失败**先修复再继续**；未通过的项**不得标记完成**（见红旗表"未自检不许声称完成"）。
3. **写「变更摘要」**到 `generate-<YYYYMMDD>.md`：改动文件清单、各自检项的结果、需人工调整的建议项、遗留风险。

## 收尾
按 `coolflow-using-flow` 纪律调用 `coolflow-precipitate` 反思沉淀
（常见可沉淀项：踩坑记录、可统一的编码规范）。

## 红旗表
| 念头 | 真相 |
|---|---|
| "方案还没评审，先写起来" | 违反 code-needs-design。停，先过方案评审。 |
| "不用列清单，我直接一把写完" | 必须先拆任务清单、再逐步执行、每步留痕。 |
| "几步并一起做更快" | 一步接一步执行，每步打标记+记录，便于追溯与回退。 |
| "先写个占位/Mock，能跑就行" | 要生产级真实代码，符合项目规范，不许占位糊弄。 |
| "项目规范我记得，不用加载" | 规范会演进。先读项目 CLAUDE.md 取索引，再渐进式披露读相关规则，别写死目录。 |
| "项目有测试设施，我也不提，省事" | 有单测设施必须问用户是否补单测；用户要则纳入任务清单。 |
| "相关测试我笼统说一句通过就行" | 步骤 4 须先列自检清单再逐项执行：lint/类型/相关测试(单测或关键逻辑校验)/需求对齐，逐项记录结果。 |
| "自检跳过，应该没问题" | 未自检不许声称完成。 |
